Introduction et problématique

1.1 Contexte et motivation

Le droit français est l'un des systèmes juridiques les plus codifiés au monde : des centaines de milliers d'articles répartis dans une quarantaine de codes officiels définissent les droits et obligations de chaque citoyen. Pourtant, cet immense corpus normatif reste largement inaccessible au grand public. Comprendre ses droits face à un licenciement abusif, un loyer impayé ou une clause contractuelle suspecte nécessite soit une formation juridique spécialisée, soit les honoraires d'un avocat — souvent prohibitifs pour les particuliers.

Des outils comme DoNotPay, LegalZoom et Rocket Lawyer ont démontré qu'il existait un besoin massif et un marché viable pour l'assistance juridique automatisée. Cependant, ces outils sont principalement conçus pour le droit américain et s'appuient sur des bases de règles rigides plutôt que sur une compréhension réelle des textes de loi.

Problématique centrale : Comment concevoir un système d'assistance juridique capable de comprendre les textes officiels français, de retrouver les articles pertinents face à une situation donnée, et de générer des réponses et documents officiels fondés exclusivement sur ces textes — sans inventer ni halluciner ?

1.2 Solution proposée : LexAI

LexAI est un assistant juridique intelligent basé sur une architecture RAG (Retrieval-Augmented Generation) hybride, spécialisé dans le droit français. La stratégie RAG a été choisie délibérément pour ses propriétés fondamentales :

  • Ancrage factuel — chaque réponse est fondée sur des articles de loi réels, cités et vérifiables sur Légifrance
  • Mise à jour sans re-entraînement — ajouter de nouveaux codes revient à étendre le corpus, pas à réentraîner le LLM
  • Traçabilité — les sources utilisées sont affichées, l'utilisateur peut vérifier lui-même
  • Contrôle des hallucinations — le LLM est contraint au contexte fourni, interdisant toute invention

1.3 Périmètre du projet

💬 Consultation juridique

📝 Rédaction de plainte

3.1 Problématique du Sprint 1

Le défi fondamental d'un système RAG juridique est le décalage de distribution entre les questions en langage naturel des utilisateurs et les textes de loi. Un article du Code Civil est rédigé dans un style impersonnel, normatif, dense — à l'opposé d'une question du type "Mon propriétaire peut-il entrer chez moi sans prévenir ?".

3.3 Choix techniques clés du Sprint 1

Recherche hybride BM25 + FAISS

Dès le Sprint 1, la décision est prise de ne pas utiliser uniquement FAISS. Le droit français utilise des termes techniques très précis (alinéa, rescision, préjudice) pour lesquels la correspondance lexicale exacte (BM25) surpasse la similarité sémantique. La fusion par Reciprocal Rank Fusion (Robertson, 1977) combine les deux signaux de façon robuste :

class HybridRetriever:
    def _rrf(self, rank: int, k: int = 60) -> float:
        return 1.0 / (k + rank)

    def invoke(self, question: str, ...) -> list[Document]:
        # FAISS : similarité sémantique (embeddings)
        vec_results = self.vs.similarity_search_with_score(query_vec, k=fetch_k)

        # BM25 : correspondance lexicale exacte
        bm25_scores = self.bm25.get_scores(question.lower().split())

        # Fusion RRF : chaque technique vote pondérément
        score[idx] += 0.65 * self._rrf(rank_faiss)   # vecteur
        score[idx] += 0.35 * self._rrf(rank_bm25)    # lexical

        return [self.documents[i] for i in top_indices]

Chunking intelligent par article

Les articles juridiques ont des longueurs très variables (quelques mots à plusieurs milliers de tokens). Un chunking uniforme briserait la cohérence juridique. La stratégie retenue :

def chunker(self, article: dict, texte_nettoye: str) -> list[Document]:
    prefixe = f"[{article['code']} — {article['article']}]\n"

    if self.compter_tokens(texte_nettoye) <= 800:
        # Article court : 1 chunk avec préfixe + domaine
        contenu = prefixe + f"Domaine : {article['domaine']}\n\n" + texte_nettoye
        return [Document(page_content=contenu, metadata={...})]

    # Article long : split récursif par alinéa avec overlap 400 chars
    morceaux = self.SPLITTER.split_text(texte_nettoye)
    return [Document(page_content=prefixe + morceau, ...) for morceau in morceaux]

Persistance FAISS intelligente

La construction de l'index FAISS prend entre 28s (modèle local) et 60s (API OpenAI). Pour éviter cette latence à chaque redémarrage, un mécanisme de persistance compare le nombre de chunks indexés au nombre actuel :

count_file = os.path.join(FAISS_PERSIST_DIR, "chunk_count.txt")
if os.path.exists(count_file):
    nb_existants = int(open(count_file).read())
    if nb_existants == len(documents):
        return FAISS.load_local(FAISS_PERSIST_DIR, embeddings, ...)  # Cache hit
# Cache miss → reindexation + sauvegarde
vs = FAISS.from_documents(documents, embeddings)
vs.save_local(FAISS_PERSIST_DIR)

4.2 Cross-Encoder Reranking (BGE-Reranker)

Principe théorique

Le retrieval initial (BM25 + FAISS) utilise des bi-encodeurs : question et document sont encodés séparément, la similarité est calculée par produit scalaire. Cette approche est rapide mais manque de précision : le modèle ne "voit" pas la question et le document ensemble.

Les cross-encodeurs concatènent question + document et les passent ensemble dans un Transformer : l'attention croisée capture les interactions fines entre chaque token de la question et chaque token du document. Précision supérieure, mais complexité O(n) — idéal pour re-scorer un petit ensemble de candidats présélectionnés.

class CrossEncoderReranker:
    def __init__(self, model_name: str = "BAAI/bge-reranker-base"):
        from sentence_transformers import CrossEncoder
        self.model = CrossEncoder(model_name)   # XLM-RoBERTa 270M params

    def rerank(self, question: str, documents: list[Document], top_k=3):
        # Paires (question, document) — le cross-encoder voit les deux ensemble
        pairs  = [(question, doc.page_content) for doc in documents]
        scores = self.model.predict(pairs)     # attention croisée
        ranked = sorted(zip(scores, documents), key=lambda x: x[0], reverse=True)
        return [doc for _, doc in ranked[:top_k]]

Pipeline complet : Top-10 hybrid → BGE-Reranker → Top-3

4.3 HyDE — Hypothetical Document Embeddings

Problème fondamental

La similarité cosinus entre une question courte ("Peut-on licencier une femme enceinte ?") et un article de loi long ("L'employeur ne saurait rompre le contrat de travail…") est structurellement faible malgré la pertinence évidente — les distributions lexicales et stylistiques sont radicalement différentes.

Solution HyDE (Gao et al., arXiv:2212.10496)

Au lieu d'embedder la question brute, le LLM génère un document hypothétique dans le style du corpus. Ce document est ensuite embedé pour FAISS. Le BM25 continue sur la question originale (complémentarité intentionnelle).

class HyDEGenerator:
    PROMPT_HYDE = """Tu es un expert en droit français. Génère un court passage (3-4 phrases)
dans le style officiel d'un article de loi français qui répondrait directement
à cette question. Utilise un langage juridique formel et précis.
Ne cite pas d'articles réels.

Question : {question}
Article hypothétique :"""

    def generer(self, question: str) -> str:
        return self.llm.invoke(self.PROMPT_HYDE.format(question=question)).content

Exemple : Question "Quelles sont les règles de protection des femmes enceintes au travail ?" → Document hypothétique généré : "Toute travailleuse en état de grossesse bénéficie d'une protection particulière contre le licenciement. L'employeur ne saurait invoquer la maternité comme motif de rupture du contrat…" → Ce document est stylistiquement proche des articles L1225-1 et L1225-5 → similarité cosinus maximisée.

Fonction de perte : MultipleNegativesRankingLoss

Dans un batch de taille N, chaque paire (query_i, positive_i) constitue un exemple positif. Les (N−1) autres positifs du batch deviennent automatiquement des négatifs in-batch. Avec batch_size=128, chaque query est comparée à 127 négatifs implicites — très efficace sans nécessiter de corpus de négatifs explicitement annotés.

from sentence_transformers import SentenceTransformer, InputExample
from sentence_transformers.losses import MultipleNegativesRankingLoss
from sentence_transformers.evaluation import InformationRetrievalEvaluator

model = SentenceTransformer("paraphrase-multilingual-MiniLM-L12-v2")
loss  = MultipleNegativesRankingLoss(model)

train_examples = [InputExample(texts=[p["query"], p["positive"]]) for p in paires]
dataloader     = DataLoader(train_examples, shuffle=True, batch_size=128)

model.fit(
    train_objectives=[(dataloader, loss)],
    epochs=25,
    warmup_steps=200,
    evaluator=InformationRetrievalEvaluator(queries, corpus, relevant_docs),
    evaluation_steps=500,
    save_best_model=True,        # meilleur checkpoint : epoch 15
    output_path="lexai-embeddings"
)

            for part in re.split(r"(\*\*[^*]+\*\*)", line):
                pdf.set_font("Helvetica", "B" if part.startswith("**") else "", 10)
                pdf.write(6, clean_text)

    return bytes(pdf.output())

8.2 Analyse du paradoxe IR vs RAGAS

Pourquoi le fine-tuning améliore le retrieval (+45%) mais pas les métriques RAGAS end-to-end ?

Les métriques IR (Accuracy@K, NDCG, MRR) mesurent si le bon article exact se trouve dans les Top-K résultats du retrieval. Les métriques RAGAS mesurent la qualité de la réponse finale produite par GPT-4o-mini.

GPT-4o-mini est un modèle extrêmement robuste : même si les 3 articles fournis ne sont pas les plus pertinents (OpenAI embeddings), il génère une réponse correcte et non-hallucinée en s'appuyant sur leur contenu partiel. C'est un effet plafond (ceiling effect) — le LLM "absorbe" les imperfections du retrieval.

Conclusion : le gain du fine-tuning est réel et mesurable (+45% Accuracy@1) mais se manifeste principalement au niveau du retrieval. Les deux approches (OpenAI vs fine-tuné) sont équivalentes en qualité de réponse finale (écart < 3% sur RAGAS). Le fine-tuning sera plus impactant avec des questions très spécifiques nécessitant l'article exact.

Difficultés rencontrées et solutions

D1 — Incompatibilité API ragas 0.4.3

La bibliothèque RAGAS a connu 3 ruptures d'API majeures entre les versions 0.1.x, 0.2.x et 0.4.x. La nouvelle API retourne des objets qui ne passent pas le test isinstance(m, Metric) de la fonction evaluate().

# ❌ INCORRECT — nouvelles classes :
from ragas.metrics.collections import Faithfulness

# ✅ CORRECT — anciens singletons (instances de Metric) :
from ragas.metrics import faithfulness, context_precision, context_recall

D2 — Answer Relevancy non calculable

La métrique answer_relevancy utilise l'interface LangChain embed_query(), supprimée dans langchain-openai >= 0.2.0. Non-bloquant — les 3 autres métriques sont calculées. La métrique a pu être calculée uniquement avec la combinaison Reranking+HyDE via l'API RAGAS directe.

D3 — Crash EvaluationResult après 40 minutes d'exécution

Le script RAGAS a exécuté 240 jobs pendant 40 minutes avant de crasher à la dernière ligne sur un accès incorrect à l'objet de résultat.

# ❌ INCORRECT :
val = result.get("faithfulness")   # EvaluationResult n'est pas un dict

# ✅ CORRECT :
df  = result.to_pandas()
val = df["faithfulness"].mean(skipna=True)
Leçon : toujours tester l'accès aux résultats sur 2-3 questions avant de lancer une évaluation longue. La sauvegarde intermédiaire pipeline_outputs.json a permis de reprendre sans réexécuter les 40 minutes.

D4 — Code de Procédure Pénale inaccessible via sandbox PISTE

L'ID LEGITEXT000006071154 ne retourne aucun article en sandbox. Le corpus v3 couvre donc 6 codes au lieu des 7 prévus. Les 6 codes disponibles couvrent les domaines les plus fréquemment consultés.

D5 — Encodage Windows CP1252 vs UTF-8

Le terminal Windows encode les print() en CP1252 par défaut. Caractères Unicode (→, accents dans f-strings) provoquent des UnicodeEncodeError. Solution : remplacement de → par -> dans tous les print(), et encodage explicite des sorties fichier en UTF-8.

D6 — Changement de dimensions FAISS (1536 → 384)

Le passage des embeddings OpenAI (1536 dims) au modèle fine-tuné (384 dims) invalide l'index FAISS existant. Solution : répertoire dédié faiss_index_v3/, variable d'environnement LEXAI_FORCE_OPENAI pour basculer sans modifier le code.

D7 — Génération PDF avec caractères hors Latin-1

Les polices core de fpdf2 (Helvetica, Times) utilisent Latin-1. Les emojis présents dans les réponses LexAI (⚖️, 📄…) provoquent des erreurs d'encodage. Solution : fonction _latin1() qui ignore silencieusement les caractères hors-range.

def _latin1(t: str) -> str:
    return t.encode("latin-1", errors="ignore").decode("latin-1")

Bilan personnel

Bilan personnel — GYEBRE Brice Joseph Emeric

Ce projet a été l'une des expériences techniques les plus enrichissantes de mon parcours académique. Partir d'un pipeline RAG de 60 articles pour arriver à un système avec fine-tuning d'embeddings, évaluation RAGAS, et génération de documents juridiques officiels en 3 sprints de 2 semaines — c'est un enchaînement qui m'a forcé à aller chercher des connaissances très au-delà du cours.

La partie qui m'a le plus surpris est le paradoxe IR vs RAGAS : le fine-tuning améliore le retrieval de +45% en Accuracy@1, mais les métriques RAGAS end-to-end restent quasi stables. Comprendre que GPT-4o-mini est suffisamment robuste pour "absorber" les imperfections du retrieval — créant un effet plafond sur les métriques de qualité finale — m'a appris à distinguer la qualité d'un composant de la qualité du système complet. Une leçon importante en ingénierie de systèmes IA.

La rigueur méthodologique imposée par le format sprint a été déterminante. Mesurer avant d'implémenter, conserver toutes les versions des datasets, documenter chaque difficulté — ces habitudes m'ont évité plusieurs catastrophes (le crash RAGAS après 40 minutes, la perte potentielle de l'index FAISS lors du changement de dimensions).

Sur le plan technique, j'ai acquis une expertise réelle dans l'écosystème RAG (LangChain, FAISS, sentence-transformers, RAGAS), dans le fine-tuning de modèles Transformer sur domaine spécialisé, et dans le déploiement d'applications IA en production (Streamlit Cloud, HuggingFace Hub). Ces compétences correspondent directement aux profils recherchés dans l'industrie en 2026.
